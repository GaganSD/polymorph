# Optimizing Coding Agent Runtimes via Latent Multi-Rubric Pruning and Lossless Context Management

## Architectural Bottlenecks of Long-Context Software Engineering Agents

Large language model powered software engineering agents automate complex, repository-level tasks such as debugging, refactoring, and feature implementation. However, as software repositories scale, these agents confront a severe processing constraint known as the context wall. Quantitative evaluations reveal that state-of-the-art coding agents spend approximately 67% to 76% of their entire token budget merely reading passive repository files retrieved during execution cycles. This massive context bloat degrades system performance through attention dilution and the lost-in-the-middle phenomenon, where critical code blocks or task-specific constraints are overlooked by the model's attention layers. Compounding this problem is the prohibitive operational cost of deploying long-context agentic loops. Modern software agents execute iterative loops containing file reading, editing, and execution commands. When a context compression system naively prunes tokens, it often introduces syntax-breaking modifications, such as dropping class headers, import statements, or matching control-flow blocks like try-except structures. When frontier models encounter such broken structures, their performance degrades, causing them to fail and execute repeated file-reading and repair operations. These repetitive loops consume more cumulative tokens than if the agent had processed the uncompressed raw context, defeating the purpose of context compression. To build a high-efficiency runtime, engineers require a deterministic, lossless prompt compressor that optimizes active context usage without losing historic conversation states. This document details the mathematical, database, and algorithmic mechanisms needed to construct a token-locking pipeline that integrates latent multi-rubric line-level pruning with database-backed, lossless context management.

## Latent Multi-Rubric Pruning Mechanics and Mathematical Formulation

Traditional learned prompt compressors, such as SWE-Pruner, formulate code pruning as a single-objective sequence labeling task. These models are trained by minimizing the conditional random field negative log-likelihood (CRF-NLL). However, this formulation introduces a severe modeling bottleneck because it uses a single CRF transition matrix to govern all facets of code relevance. Code relevance is inherently multi-dimensional, exhibiting distinct, conflicting sequence patterns : - Semantic Evidence: This dimension represents the topical relevance of code blocks to the user's natural language query. Semantic relevance tends to form contiguous, high-density blocks of code, requiring a transition matrix with high self-transition probabilities.

- Dependency Support: This dimension captures the structural scaffolding lines—such as import statements, class declarations, function signatures, and enclosing control-flow statements—that are necessary to compile and interpret the semantic blocks. Dependency lines are highly sparse and isolated, requiring a completely different transition matrix. The Latent Multi-Rubric (LaMR) framework resolves this conflict by modeling code relevance through $K$ separate latent rubrics, each governed by an independent CRF head with its own transition matrix.

In practice, setting $K=2$ allows the system to isolate semantic evidence from dependency support.

### Mathematical Formulation

Given a code context sequence $C = (x_1, \dots, x_n)$ of $n$ tokens and a natural-language goal hint query $q$ describing the agent's current task, the goal is to predict a binary sequence $\mathbf{y} \in \{\text{keep}, \text{prune}\}^n$. The model processes the input through a dual-CRF architecture. For each latent rubric $k \in \{1, \dots, K\}$, a dedicated head computes a sequence of rubric-specific emission potentials $\mathbf{E}^k(i) \in \mathbb{R}^2$ for each token $x_i$, alongside a dedicated transition matrix $\mathbf{A}^k \in \mathbb{R}^{2 \times 2}$. To dynamically balance these rubrics based on the active task, a query-adaptive Mixture-of-Experts (MoE) gating network calculates a weighting coefficient $g_k(q)$ for each rubric :

$$g_k(q) = \frac{\exp(\mathbf{w}_k^\top \mathbf{h}_q)}{\sum_{j=1}^K \exp(\mathbf{w}_j^\top \mathbf{h}_q)}$$

where $\mathbf{h}_q$ represents the dense embedding vector of the natural language query $q$, and $\mathbf{w}_k$ represents the learned parameter weights for the $k$-th gate. The dynamic per-rubric emissions are fused using these gating coefficients to produce a combined emission matrix :

$$\mathbf{E}(i) = \sum_{k=1}^K g_k(q) \mathbf{E}^k(i)$$

The final sequence-level classification is generated by a master CRF layer operating on these fused emissions with a global transition matrix $\mathbf{A} \in \mathbb{R}^{2 \times 2}$. The conditional probability of a target label sequence $\mathbf{y}$ is defined as:

$$P(\mathbf{y} \mid C, q) = \frac{1}{Z(C, q)} \exp\left( \sum_{i=1}^n \mathbf{E}(i)_{y_i} + \sum_{i=1}^{n-1} \mathbf{A}_{y_i, y_{i+1}} \right)$$

where $Z(C, q)$ represents the partition function normalizing the probability over all possible paths. During training, the parameters of the encoder, the MoE gating network, and the multiple CRF heads are optimized jointly by minimizing the global CRF negative log-likelihood (CRF-NLL). At inference time, the optimal path is recovered using Viterbi decoding.

## Hybrid Semantic-Structural Evaluation: Static Analysis versus Machine Learning

A key architectural design question is whether the multi-rubric model relies on machine learning or static analysis to evaluate code structure and natural language prose. The LaMR framework implements a hybrid execution model that cleanly decouples training-time label generation from runtime inference:

| Execution Phase | Evaluation Mechanism | Input Modalities | Computational Cost | Primary Purpose |

| --- | --- | --- | --- | --- |

| Training Phase | Static Program Analysis (AST parsing, dependency graphs) | Source code files, teacher-retained binary masks | Low (Zero LLM API cost, static AST lookups) | Denoising noisy teacher masks, generating multi-rubric labels |

| Inference Phase | Pure Machine Learning (0.6B neural skimmer, MoE gate, CRF heads) | Natural language query $q$, raw retrieved code context $C$ | High (GPU forward pass, Viterbi decoding) | Generating real-time line-level keep-or-prune decisions |

## Zero-Cost Rubric-Guided Labeling via AST Parsing

During training, the system faces a cold-start problem: existing software engineering datasets only provide raw, binary keep-or-prune masks $Y_{\text{teacher}}$ produced by a teacher LLM, without rubric-specific labels. To supervise the dual-CRF heads without incurring additional LLM annotation costs, LaMR uses static program analysis via an Abstract Syntax Tree (AST) parser to derive clean, multi-dimensional labels from the binary teacher masks. The Rubric-Guided Labeling pipeline processes code contexts through two parallel steps:

- Semantic Labeling ($y^{\text{semantic}}$): Lines retained in the teacher's original binary mask are assigned the document-level relevance score, while discarded lines are set to zero. Dependency Labeling ($y^{\text{dependency}}$): Using the AST, the system traces dependency edges starting from the semantic lines. These edges include symbol definitions, module imports, function call sites, and parent scope headers. The dependency score $S_{\text{dep}}(v)$ for any AST node $v$ is computed by propagating scores over $h$ hops using a geometric decay formula :

$$S_{\text{dep}}(v) = \max_{u \in Y_{\text{teacher}}} \gamma^{d(u, v)}$$

where $d(u, v)$ is the shortest path distance (number of hops) between the seed node $u$ and target node $v$ in the dependency graph, and $\gamma \in (0, 1)$ is a decay factor (typically set to $0.8$). Furthermore, to maintain syntactic coherence, the pipeline parses compound statements (such as try, if, with, or for blocks). If a line inside one of these blocks is retained, the labeling pipeline automatically assigns a dependency score of $1.0$ to the block header and its sibling branches (e.g., except, finally, or else clauses). The final dependency score vector is binarized using a threshold of $0.5$ :

$$y_i^{\text{dependency}} = \mathbb{I}(S_{\text{dep}}(v_i) \ge 0.5)$$

At inference time, the system bypasses this static AST parsing completely. The 0.6B-parameter neural skimmer evaluates both the natural language prose and the structural code blocks purely using its learned neural weights, predicting line-level keep-or-prune sequences based on internal latent representations.

## Lossless Context Management versus Stochastic Window Compaction

While in-place code pruning optimizes file-reading operations, the overall execution history must also be managed. Traditional context compression systems and native model contexts (such as Claude's native context window) rely on sliding-window mechanisms or lossy flat summarization. This comparison highlights the structural trade-offs between native model context management, Recursive Language Models (RLMs), and Lossless Context Management (LCM) :

| Architectural Attribute | Claude Native Context | Recursive Language Models (RLM) | Lossless Context Management (LCM) |

| --- | --- | --- | --- |

| State Persistence | Transient active window; old turns are deleted or summarized in a lossy flat structure. | Volatile session state written to a Read-Eval-Print Loop (REPL). | Transactional, database-backed, append-only Immutable Store. |

| Control Layer | Stochastic model-authored chunking and rollup scripts. | Stochastic; model generates symbolic code to recursively call and update itself. | Deterministic engine runtime enforcing strict soft and hard token thresholds. |

| Context Recovery | Irrecoverable; once compacted, character-level details are permanently lost. | Re-execution of model-authored code loops, which are prone to failure and infinite loops. | Deterministic, explicit lineage pointers; drill-down using lcm_expand and lcm_grep. |

| Failure Protection | None; context saturation triggers model failures, hallucinations, or repetitive loops. | Vulnerable to syntax errors, execution exceptions, and logic drift in generated scripts. | Guaranteed convergence via a Three-Level Escalation protocol with deterministic truncation fallback. |

| Execution Latency | High overhead during summarization spikes, interrupting user interaction. | High overhead due to recursion logic executed on every conversational turn. | Low; compaction passes run asynchronously between turns with zero user-perceived delay. |

## Dolt Retrieval Traversal: Direct Memory Recovery

To recall specific details from compacted history without saturating the active context window, the agent uses a deterministic traversal mechanism called Dolt Retrieval Traversal:

```
[Main Agent Active Context]
|
+---> Memory Cue (Summary ID + Lineage Pointer)
|
v [lcm_describe(node_id)] ---> Manifest of Off-Context Nodes & Lineage Closure
|
v [lcm_expand(node_id)] ---> Traces lineage -> Retrieves Verbatim Logs from Database
```

First, pre-response hooks automatically inject memory cues into the top of the active context window. These cues contain summary IDs, summary type metadata, and lineage pointers, referencing either an active bindle or an archived archive_stub pointer. Next, when the agent needs to inspect a historic turn, it calls lcm_describe. This tool returns detailed metadata, off-context statuses, target nodes, and lineage closure IDs without pulling the raw content into the prompt. Finally, the agent invokes lcm_expand on the target node. This tool traces the lineage mapping, retrieves the raw messages from the database, and loads them into a temporary, isolated sub-agent context window. The sub-agent extracts the exact technical details and returns them to the main agent, keeping the primary prompt compact and highly focused.

## Directed Acyclic Graph Summary Trees and Deterministic Control Loops

As a conversation scales, LCM structures summary nodes into a multi-resolution, Directed Acyclic Graph (DAG). This structure ensures that older conversation segments are represented at increasing levels of abstraction, while maintaining explicit lineage links back to the original source text.

```
                    +---------------------+---------------------+
                    v                     v
              +------+------+       +------+------+
              v      v      v       v      v      v
         (Turns 1-4) ... (Turns 13-16)
              |      |      |       |      |      |
              +------+------+       +------+------+
                    v                     v
              Depth-0 summaries     Depth-1 / Depth-2 summary tree
```

This multi-level summary tree is constructed through a series of automated passes:

- Depth-0 Summaries (Local Specifics): When raw messages outside the protected fresh tail exceed a set token threshold (typically 2,000 tokens), they are summarized into a Depth-0 node. These summaries capture precise technical decisions, code changes, and execution results.
- Depth-1 Summaries (Span Synthesis): When a set number of Depth-0 nodes accumulate (typically 4 nodes), they trigger a condensation pass. These are synthesized into a single Depth-1 summary that distills the broader evolution of the task.
- Depth-2 summaries (Global Abstract): As Depth-1 summaries accumulate, they are compacted into highly abstract Depth-2 nodes. This structure allows the agent to inspect the overall session arc and expand only the specific subtrees relevant to its current step.

## Deterministic Control Loop and Escalation Protocol

To maintain system reliability, LCM bypasses the model to handle context limits. A deterministic control loop monitors active context tokens against soft ($\tau_{\text{soft}}$) and hard ($\tau_{\text{hard}}$) thresholds. If active tokens remain below $\tau_{\text{soft}}$, the engine appends messages normally with zero operational overhead. When context usage crosses $\tau_{\text{soft}}$, the engine compacts older messages asynchronously and swaps the resulting summary into the active context between turns. If a compaction pass fails to reduce token usage (e.g., the model generates an overly long summary), the engine triggers a three-level escalation protocol to guarantee convergence :

| Escalation Level | Execution Mechanism | Operational Parameters | Primary Invariant |

| --- | --- | --- | --- |

| Level 1 (Normal) | Detail-preserving LLM summary. | Summarizes older nodes using a detailed preserve_details prompt with target size $T$. | Captures specific technical decisions and implementation details. |

| Level 2 (Aggressive) | High-compression LLM summary. | Drops reasoning chains; generates a high-density, bulleted summary restricted to $T / 2$ tokens. | Drops conversational prose to prioritize raw technical metrics and key facts. |

| Level 3 (Deterministic) | Truncation with zero model inference. | Hard-crops the older text block to exactly 512 tokens. | Bypasses LLM calls completely to guarantee prompt convergence. |

## Operator-Level Recursion and the Infinite Delegation Guard

Instead of letting models generate stochastic loops to manage long-context tasks, LCM introduces Operator-Level Recursion via two engine-managed tools : - LLM-Map: This tool applies an LLM prompt in parallel across items in a JSONL file. The engine manages a worker pool (defaulting to 16 parallel threads), handles rate limits, validates outputs against a JSON Schema, and processes structured retries. Because each run is a stateless, pure function, the active context remains clear of intermediate orchestration logs. - Agentic-Map: This tool spawns a full sub-agent session for each item in a list. Sub-agents are granted access to file I/O and code execution tools, allowing them to solve complex, multi-step engineering tasks in isolated environments. To prevent nested sub-agents from looping infinitely, the engine enforces a scope-reduction invariant. When spawning a child agent, the parent must declare both the delegated scope and the remaining work. The engine analyzes these inputs and rejects the delegation if the child's scope is not a strict subset of the parent's task, or if the remaining work definition is empty. This enforces convergence down a well-founded recursion tree without relying on arbitrary depth limits.

## Runtime Integration Bottlenecks, Token Estimation Bugs, and Remediation

Integrating learning-based pruners and database-backed context engines into a single runtime introduces several critical, real-world failure modes. Our token-locking pipeline must implement specific fixes to resolve these issues:

1. CJK and

Emoji-Aware Token Estimation Underestimation — A major bug in the reference implementation of lossless-claw is its token estimation function, which uses Math.ceil(text.length / 4). While this heuristic is accurate for standard English text (~4 ASCII characters per token), it severely underestimates token counts by 2x to 4x for CJK (Chinese, Japanese, Korean) and emoji-heavy text, where each character maps to ~1.5 tokens in modern tokenizers (e.g., cl100k_base or o200k_base). This underestimation leads to several runtime failures: compaction passes trigger too late, causing active context window overflows and API failures; context assembly budgets are miscalculated, truncating active file contents; the large-file interception boundary fails to capture CJK-heavy files, injecting massive payloads that saturate the prompt. To fix this, our pipeline must replace this simple length-based heuristic with a unified, CJK-aware token estimation utility. This module uses a fast, local tokenizer library to calculate exact token counts across diverse scripts, and includes an idempotent migration script to correct historical token counts in the database.

2. Silent

Failures under OpenAI-Compatible Gateways — In production runs using lossless-claw with OpenAI-compatible gateways, the context engine's afterTurn handler can silently fail. The gateway only provides a tokenBudget when context usage can be calculated from native session metadata (e.g., for Anthropic models). For OpenAI-compatible endpoints or local LLM proxies, the gateway passes undefined for the token budget. As a result, the afterTurn handler silently bails out. Messages continue to accumulate in the database with zero summaries, and because the system logs no errors, users assume the context engine is working while their active prompt token counts climb unchecked. Our token-locking pipeline resolves this by implementing local token accounting at the engine level. The runtime uses a local tokenizer to track the active context's token size and trigger compaction passes independently of gateway-supplied metadata.

3. Pipeline

Configuration and Optimization Parameters — To optimize database storage, context-assembly speeds, and execution latencies, the token-locking pipeline uses a series of performance-tuning parameters:

| Configuration Key | Default Value | Recommended Token-Locking Setting | Technical Impact on Pipeline Behavior |

| --- | --- | --- | --- |

| LCM_CONTEXT_THRESHOLD | 80% of native window. | 120000 (for 128K models) / 800000 (for 1M models). | Decides when compaction starts; lower values build the summary DAG earlier, saving active prompt space at the cost of more frequent API calls. |

| LCM_FRESH_TAIL_COUNT | 4 messages. | 10 to 15 messages. | Protects recent messages from compaction, ensuring the agent retains raw context for recent tool calls and execution logs. |

| LCM_LEAF_CHUNK_TOKENS | 20000 tokens. | 16000 tokens. | Controls the raw text size for Depth-0 summaries; reducing this value generates more detailed leaf summaries. |

| LCM_NEW_SESSION_RETAIN_DEPTH | 2 (Depth-2 summaries). | 2. | Controls how much summary structure survives session resets (/new); setting this to 2 keeps high-level abstract summaries while dropping raw tail messages. |

| LCM_SKIP_STATELESS_SESSIONS | false. | true. | Skips persistence and compaction for bootstrap imports, cron sessions, and stateless tasks, reducing database write bottlenecks. |

| LCM_LARGE_OUTPUT_EXTERNALIZATION_ENABLED | false. | true. | Detects large tool outputs, logs, or JSON payloads, automatically saving them as external database files and injecting a compact handle and summary into the active context. |

## Blueprint for a Unified Token-Locking Pipeline

Integrating the line-level pruning of LaMR with the transactional context engine of LCM creates a highly optimized, deterministic, and lossless prompt compressor. The system processes incoming data and maintains state through a unified runtime loop:

```
[ Conversational History ]          [ Active Code Retrieval ]
| |
v                                    v LCM compaction                      LaMR pruning (Dual CRFs)
| |
+----------------+-------------------+ v [ Optimized Active Prompt ] ---> Target LLM
```

1. Active

Code Retrieval Pipeline (LaMR-Core) — When the agent executes a file-reading tool (e.g., cat or grep), the raw context is intercepted before entering the active context window. The raw file content and the active natural language query are sent to the 0.6B neural skimmer. The model computes parallel semantic and dependency emission potentials, fuses them using the query-adaptive gating weights, and applies Viterbi decoding to generate line-level keep-or-prune decisions. To guarantee syntax safety, the pruned lines are sent to a local, lightweight AST validation module. If the skimmer retains a line inside a class, function, or compound control block but discards its header or imports, this validation module automatically restores those required structural lines before sending the code block to the agent, preventing compilation failures and retry loops.

2. Conversational

History Pipeline (LCM-Core) — Every user prompt, assistant response, and raw tool output is logged verbatim to a local SQLite or PostgreSQL database, preserving a complete, append-only history of the session. The engine tracks active context size using a CJK-aware token estimation library. When the active context crosses the soft threshold ($\tau_{\text{soft}}$), the engine triggers an asynchronous background worker to summarize older message blocks outside the protected fresh tail, swapping in the new summary node before the next turn. If token usage exceeds the hard threshold ($\tau_{\text{hard}}$), the engine runs the Three-Level Escalation protocol. This escalates from detail-preserving summaries to high-compression bullet points, and finally falls back to deterministic truncation to guarantee context window convergence. When the agent needs to access historical details, it uses lcm_grep and lcm_describe to locate the target summary node and runs lcm_expand to retrieve and process the raw message block in an isolated sub-agent context window. By offloading context orchestration to a deterministic database engine and using a lightweight, structure-aware model to prune source files, this token-locking pipeline minimizes token overhead, prevents reasoning degradation, and maintains absolute historical recall across long agentic sessions.
